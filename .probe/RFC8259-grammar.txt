# RFC 8259 — JSON Data Interchange Format (Bray, 2017)
# Fetched via mcp__rfc__fetch_rfc(8259) on 2026-06-18.
# Below: the ABNF grammar (Sections 2-7) + interop MUSTs the parser/serializer enforces.

# --- ABNF (Section 2) ---
JSON-text = ws value ws
ws = *( %x20 / %x09 / %x0A / %x0D )

# --- Values (Section 3) ---
value = false / null / true / object / array / number / string
false = %x66.61.6c.73.65   ; "false"
null  = %x6e.75.6c.6c      ; "null"
true  = %x74.72.75.65      ; "true"

# --- Objects (Section 4) ---
object = begin-object [ member *( value-separator member ) ] end-object
member = string name-separator value
begin-object    = ws %x7B ws       ; {
end-object      = ws %x7D ws       ; }
name-separator  = ws %x3A ws       ; :
value-separator = ws %x2C ws       ; ,

# Names within an object SHOULD be unique. Behaviour on duplicates is
# implementation-defined; we keep the LAST occurrence (matches most libs).
# Member ordering is NOT preserved by spec but we PRESERVE it for serializer
# determinism (helps reproducible tests + canonical hashing).

# --- Arrays (Section 5) ---
array = begin-array [ value *( value-separator value ) ] end-array
begin-array = ws %x5B ws            ; [
end-array   = ws %x5D ws            ; ]

# --- Numbers (Section 6) ---
number        = [ minus ] int [ frac ] [ exp ]
decimal-point = %x2E
digit1-9      = %x31-39
e             = %x65 / %x45
exp           = e [ minus / plus ] 1*DIGIT
frac          = decimal-point 1*DIGIT
int           = zero / ( digit1-9 *DIGIT )
minus         = %x2D
plus          = %x2B
zero          = %x30
# Leading zeros NOT allowed. Infinity / NaN NOT permitted.
# IEEE 754 double is the interop-safe range; we store inline Int64 or Float64
# and spill to BIG_NUMBER (raw text) when neither fits — caller reads via
# Value.as_str_number().

# --- Strings (Section 7) ---
string         = quotation-mark *char quotation-mark
char           = unescaped / escape (
                    %x22 /              ; "    quotation mark   U+0022
                    %x5C /              ; \    reverse solidus  U+005C
                    %x2F /              ; /    solidus          U+002F
                    %x62 /              ; b    backspace        U+0008
                    %x66 /              ; f    form feed        U+000C
                    %x6E /              ; n    line feed        U+000A
                    %x72 /              ; r    carriage return  U+000D
                    %x74 /              ; t    tab              U+0009
                    %x75 4HEXDIG )      ; uXXXX                 U+XXXX
escape         = %x5C                   ; \
quotation-mark = %x22                   ; "
unescaped      = %x20-21 / %x23-5B / %x5D-10FFFF
# Control chars 0x00..0x1F MUST be escaped (cannot appear bare).
# Section 7: ">U+FFFF" uses 12-char surrogate-pair encoding `𝄞`.
# Section 8.2 acknowledges unpaired surrogates produce undefined behaviour —
# our parser accepts them (lossy) and our serializer emits them as raw \uXXXX
# without revalidating. M1: a strict-mode flag to reject.

# --- Encoding (Section 8.1) ---
# JSON text exchanged between systems MUST be encoded using UTF-8.
# MUST NOT add a BOM on emit; MAY ignore BOM on parse.
# Our parser: silently skips a leading 0xEF 0xBB 0xBF.

# --- Parser conformance (Section 9) ---
# MUST accept all texts that conform to the grammar.
# MAY set limits on size / nesting depth / number range / string length.
# Our limits: default unlimited; v1 may add a `ParseConfig` with max_depth /
# max_string_len / max_number_digits.

# --- Generator conformance (Section 10) ---
# Output MUST strictly conform to the JSON grammar.
